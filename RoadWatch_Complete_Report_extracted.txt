053 [Heading1] 1. INTRODUCTION
054 [] RoadWatch is an AI-powered, cross-platform mobile application developed using Flutter (frontend) and Django REST Framework (backend). It addresses a critical infrastructure problem faced by rapidly urbanising cities across India: the inability to detect, report, and resolve road damage in a timely and efficient manner. Traditional complaint mechanisms suffer from 3–7 day response times, manual inspection bottlenecks, no real-time citizen engagement, and data silos between citizens and government authorities.
055 [] RoadWatch bridges this gap by combining mobile AI with community-driven crowdsourcing. Citizens can photograph a road issue — be it a pothole, crack, flooding, debris, or damaged signage — and submit a GPS-tagged report in under 45 seconds. The AI-powered backend classifies the issue by type and severity, while the community verification system allows other citizens to upvote or downvote a report to validate its authenticity. A dedicated Government/Admin Dashboard provides real-time analytics and enables data-driven resource allocation.
056 [] The application is built as a full-stack system: the Flutter frontend targets Android 8.0+, iOS 14+, and Web PWA; the Django 5.0 REST API handles authentication, report management, notifications, and admin functions; SQLite serves as the development database; and Google Maps SDK powers the geolocation and mapping features. JWT-based authentication ensures secure, stateless sessions across all platforms.

094 [Heading1] 6. PROBLEM DEFINITION
095 [] India's rapidly urbanising cities face a severe and persistent infrastructure challenge: deteriorating road conditions that cause thousands of accidents annually, vehicle damage, economic losses, and reduced quality of life. The core problem is the absence of an efficient, real-time communication channel between citizens who observe road damage and the government authorities responsible for repair.
096 [] Existing complaint mechanisms (phone calls, offline forms, manual inspection) are slow, undocumented, geographically imprecise, and lack any community validation. A pothole reported via a helpline may take 3–7 business days to receive even an initial response, while the damage continues to pose a safety risk. Furthermore, without GPS tagging, reports are difficult to locate accurately, leading to miscommunication between field workers and administrators.
097 [] RoadWatch solves this problem by providing: instant mobile reporting with GPS precision, AI-driven issue classification and severity rating, community crowdsourcing for report verification, and a government dashboard for data-driven repair scheduling.

148 [Heading1] 13. MODULE DESCRIPTION
149 [] RoadWatch is organised into the following modules, each corresponding to a Flutter screen and/or a set of Django API views:
150 [Heading2] Module 1: Authentication Module
151 [] Files: LoginScreen.dart, registration_screen.dart, forgot_password_screen.dart, reset_password_screen.dart, auth_service.dart, storage_service.dart
152 [] This module handles the complete user authentication lifecycle. Users register with email and password; on successful registration, a JWT access and refresh token pair is returned and stored securely via flutter_secure_storage. Login authenticates credentials against the Django backend; role (admin/user) is stored and used for routing. The module also supports forgot password and reset password flows. Demo mode allows exploration without credentials.
153 [Heading2] Module 2: Home / Dashboard Module
154 [] Files: home_screen.dart, splash_screen.dart, api_service.dart
155 [] The dashboard is the main entry point post-login. It fetches real-time statistics from /api/dashboard/ (total reports submitted, verified reports, upvotes received) and displays them as summary cards. Recent reports are listed as scrollable cards with issue type, location, date, severity badge, and status chip. A floating action button navigates to the Report submission screen. A bottom navigation bar provides access to all five main screens.
156 [Heading2] Module 3: Report Submission Module
157 [] Files: report_screen.dart, services/api_service.dart, services/google_maps_loader.dart
158 [] This is the core module. Users select an issue type (Pothole, Crack, Flooding, Debris, Signage, Other), add a description, choose severity (Low/Medium/High), capture or upload a photo via image_picker, and submit with GPS tagging. The Google Maps widget shows the current detected location with a draggable pin for adjustment. Reverse geocoding converts coordinates to a human-readable address using the geocoding package. On submission, a multipart POST request is sent to /api/reports/. The entire flow takes under 45 seconds.
159 [Heading2] Module 4: Reports Module
160 [] Files: reports_screen.dart, issue_detailed_screen.dart, report_print_screen.dart
161 [] The Reports screen lists all reports submitted by the logged-in user with filtering by status. Tapping a report opens the Issue Details screen showing full information: issue type, severity badge, status, location, date, upvote count, verify count, and the submitted photo. Users can upvote from this screen. A print/export button generates a PDF report using the pdf + printing packages.
162 [Heading2] Module 5: Alerts / Notifications Module
163 [] Files: alerts_screen.dart
164 [] Fetches the user's notification list from /api/notifications/ and displays them with read/unread indicators. Notifications are triggered by backend events such as report status changes. The screen auto-refreshes and allows marking notifications as read.
165 [Heading2] Module 6: Profile Module
166 [] Files: profile_screen.dart
167 [] Displays user account information (email, report count, verified count, upvote count) fetched from the profile API. Provides theme toggle (Light/Dark/System), edit profile, settings, help & support, and logout functionality.
168 [Heading2] Module 7: Admin Dashboard Module
169 [] Files: admin_dashboard.dart, backend/accounts/views.py (admin views)
170 [] Accessible only to users with role='admin'. Displays all reports from all users with full CRUD capabilities: view details, update status (Pending/Verified/In Progress/Resolved/Rejected), add admin notes, and delete reports. Also provides user management: list all users, update user details, delete users.
171 [Heading2] Module 8: Theme Module
172 [] Files: theme/app_theme.dart, theme/theme_provider.dart, theme_manager.dart
173 [] Implements a complete light/dark/system theme system using Flutter's ThemeProvider with ChangeNotifier. Theme preference is persisted via shared_preferences. Supports smooth animated transitions between themes.

548 [Heading1] 18. SECURITY MECHANISM
549 [Heading2] 18.1 Authentication Security
550 [ListParagraph] JWT (JSON Web Tokens) — All protected API endpoints require a valid Bearer token. Tokens issued on login and expire after 24 hours.
551 [ListParagraph] Bcrypt Password Hashing — All passwords stored as bcrypt hashes with minimum 10 rounds. Plain-text passwords are never stored.
552 [ListParagraph] Secure Token Storage — JWT tokens stored using flutter_secure_storage (Android Keystore / iOS Keychain hardware-backed encryption).
553 [Heading2] 18.2 Authorization & Access Control
554 [ListParagraph] Role-Based Access Control (RBAC) — User model includes a 'role' field (admin/user). Admin-only endpoints return 403 Forbidden for unauthorized access.
555 [ListParagraph] Object-Level Permission — Regular users can only DELETE/modify their own reports; admins can access any report.
556 [Heading2] 18.3 Data Security
557 [ListParagraph] CORS Protection — django-cors-headers configured; production requires explicit CORS_ALLOWED_ORIGINS whitelist.
558 [ListParagraph] Input Validation — DRF serializers validate all incoming data; Django ORM prevents SQL injection via parameterised queries.
559 [ListParagraph] Image Upload Safety — Pillow validates uploaded files are genuine images; non-image files are rejected.

576 [Heading1] 21. LIMITATIONS AND FURTHER ENHANCEMENTS
577 [Heading2] Current Limitations
578 [ListParagraph] Database: SQLite used in development is not suitable for high-concurrency production environments; migration to PostgreSQL is required for deployment.
579 [ListParagraph] AI Classification: The current version relies on user-selected issue types; automated AI image classification is planned but not yet implemented.
580 [ListParagraph] Push Notifications: In-app notifications are implemented; true background push notifications (FCM/APNs) require additional backend configuration.
581 [ListParagraph] Offline Reporting: Reports cannot be submitted without an internet connection; offline queuing is not yet implemented.
582 [ListParagraph] Map Performance: Google Maps on Android uses Hybrid Composition (required fix applied) which may consume more memory on low-end devices.
583 [ListParagraph] Admin Dashboard: The admin dashboard is mobile-only; a dedicated web-based admin portal would improve usability for government officials.
584 [ListParagraph] Password Reset: Current implementation is a demo stub; production requires email SMTP configuration (SendGrid, Mailgun, etc.).
